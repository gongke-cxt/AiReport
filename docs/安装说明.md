# AiReport 安装说明

本文档面向第一次把 `AiReport` 拉到本地的开发者，重点说明：

- 仓库里包含哪些内容
- 哪些内容不会被 Git 提交
- 拉取后需要执行哪些命令
- 当前项目哪些部分可直接运行，哪些部分仍是骨架

## 1. 先说明一个关键点

GitHub 仓库里保存的是**源码**，不是你本机已经安装好的运行环境。

因此，别人拉下来和你本机看起来“不一样”，通常不是少推代码，而是因为以下内容本来就不会提交：

- `node_modules/`
- `dist/`
- `services/api/dist/`
- `.env`
- 本机数据库、Redis、缓存和编译产物

这些内容都需要拉取代码后在本地重新生成。

## 2. 环境要求

建议准备以下环境：

- Git
- Node.js `18+`
- npm
- PostgreSQL `14+`，如果要跑 API
- Redis，如果要跑 API

当前仓库根 `package.json` 要求的 Node 版本为：

```text
>=18
```

## 3. 拉取项目

```bash
git clone https://github.com/gongke-cxt/AiReport.git
cd AiReport
```

## 4. 安装根依赖

根仓库是一个 monorepo，但当前只把下面两类包纳入了根 `workspaces`：

- `apps/*`
- `packages/*`

因此先在根目录执行：

```bash
npm install
```

这一步会安装：

- `apps/web`
- `packages/dsl-schema`
- `packages/dsl-types`

不会安装：

- `services/api`

因为 API 服务目前不在根 `workspaces` 里。

## 5. 启动前端

前端是当前仓库里最容易直接跑起来的部分。

在根目录执行：

```bash
npm run dev
```

默认访问地址：

```text
http://localhost:5173
```

可用的辅助命令：

```bash
npm run typecheck
npm run validate:dsl
```

含义如下：

- `npm run typecheck`：检查前端和 DSL 类型
- `npm run validate:dsl`：校验 DSL 样例 JSON 是否符合 Schema

## 6. API 服务单独安装

如果你还需要尝试启动 API，需要进入 `services/api` 单独安装依赖：

```bash
cd services/api
npm install
```

然后准备环境变量：

```bash
copy .env.example .env
```

或手动创建 `.env`，至少配置：

- `PORT`
- `DATABASE_URL`
- `REDIS_URL`
- `AI_PROVIDER`
- `AI_API_KEY`
- `AI_MODEL`
- `JWT_SECRET`

## 7. 初始化数据库

如果要运行 API，建议先创建数据库并执行迁移脚本。

示例流程：

```bash
psql -U postgres -c "CREATE DATABASE aireport"
psql -U postgres -d aireport -f database/migrations/001_init.sql
psql -U postgres -d aireport -f database/migrations/002_seed.sql
```

说明：

- `001_init.sql` 会创建租户、用户、项目、文档、模板、AI 会话等表
- `002_seed.sql` 会插入开发环境默认租户、默认用户和一个内置模板

默认租户和用户会配合 API 中的这两个开发参数：

- `DEFAULT_TENANT_ID`
- `DEFAULT_USER_ID`

## 8. 启动 API

进入 API 目录后执行：

```bash
npm run dev
```

理论默认端口：

```text
http://localhost:3000
```

健康检查接口：

```text
GET /health
```

## 9. 当前 API 的真实状态

这里要明确说明，当前仓库中的 API 还不是完整可运行状态。

原因有两类：

### 9.1 工程层面

- `services/api` 不在根 `workspaces`
- 需要手动进入目录执行 `npm install`

### 9.2 代码层面

`services/api/src/app.ts` 当前引用了以下路由：

- `./routes/query.js`
- `./routes/publish.js`
- `./routes/templates.js`

但这些文件目前并不存在。

因此即使你安装了 API 依赖，当前仓库的 API 也仍然是“骨架状态”，不能视为完整后端。

## 10. 推荐的本地验证顺序

如果你只是想确认仓库是否完整，建议按下面顺序执行：

```bash
git clone https://github.com/gongke-cxt/AiReport.git
cd AiReport
npm install
npm run typecheck
npm run validate:dsl
npm run dev
```

这套流程对应当前仓库已经可验证的范围：

- 前端工程骨架
- DSL 类型
- DSL Schema

如果你要继续尝试 API，再额外执行：

```bash
cd services/api
npm install
```

但需要接受当前 API 仍未补齐的事实。

## 11. 为什么“别人拉下来后不一样”

这是最常见的几个原因：

### 11.1 没有 `node_modules`

仓库不会提交依赖目录。拉下来后必须执行：

```bash
npm install
```

### 11.2 没有 `.env`

环境变量文件不会提交。需要自己根据 `.env.example` 新建。

### 11.3 没有 `dist`

构建产物不会提交。需要自己执行构建命令生成。

### 11.4 API 不是完整成品

即使代码仓库完整，当前后端仍然缺少部分路由实现，所以“不能直接跑起来”不代表“少推了文件”。

## 12. 建议给协作者的最短说明

如果你只想给别人一段最短操作说明，可以直接发下面这段：

```bash
git clone https://github.com/gongke-cxt/AiReport.git
cd AiReport
npm install
npm run typecheck
npm run validate:dsl
npm run dev
```

再补一句：

```text
仓库只包含源码，不包含 node_modules、dist、.env；API 目录需要单独 npm install，且当前后端仍是骨架。
```
