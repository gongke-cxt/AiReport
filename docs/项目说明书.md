# AiReport 项目说明书

## 1. 文档目的

本文档基于当前仓库代码与目录结构编写，目标是说明 `AiReport` 项目的：

- 产品定位与当前范围
- 代码结构与模块职责
- 前后端、DSL、数据库之间的关系
- 本地运行方式
- 当前已实现能力与待补齐缺口

文档编写时间：2026-03-29。

## 2. 项目概述

`AiReport` 是一个面向报表与大屏场景的 AI 智能报表平台原型仓库。当前仓库更接近“方案验证骨架”，而不是完整可上线产品。

从代码现状看，这个仓库主要包含四部分：

- 一个基于 React + Vite 的前端骨架，用于展示平台工作台与交互入口。
- 一组面向报表/大屏的 DSL 定义，包括 JSON Schema、TypeScript 类型和样例文档。
- 一个基于 Express + PostgreSQL + Redis 设计的 API 服务骨架。
- 一套 PostgreSQL 初始化脚本，用于承载租户、项目、文档、模板、AI 会话和查询审计等数据。

项目设计方向明显是：

- 用自然语言驱动报表/大屏生成
- 用统一 DSL 连接 AI 输出、编辑器和运行时
- 用多租户、数据源连接、模板市场和语义层支撑企业级交付

但当前实际完成度仍集中在“结构设计”和“核心模型定义”，而非完整产品功能。

## 3. 技术栈与工程形态

### 3.1 根仓库

- 包管理：`npm`
- 工程形态：monorepo
- Node 要求：`>=18`
- 当前根工作区：`apps/*`、`packages/*`

注意：`services/api` 目录存在，但当前没有被纳入根 `workspaces`，因此根目录执行 `npm install` 不会自动安装 API 服务依赖。

### 3.2 前端

- React 18
- TypeScript
- Vite 5

前端目前是一个静态骨架页面，主要作用是展示平台入口和开发占位信息，不是完整的报表编辑器或运行时。

### 3.3 后端

- Express
- TypeScript
- PostgreSQL
- Redis
- Zod

后端当前是 API 骨架。已有的路由实现只覆盖了一部分接口，仍存在依赖未安装、路由文件缺失、与 OpenAPI 草案不完全一致等问题。

### 3.4 DSL 与协议层

- JSON Schema Draft 2020-12
- TypeScript 手写类型
- AJV 校验
- OpenAPI 3.1

这一部分是当前仓库里最完整、最清晰的能力层。

## 5. 核心模块说明

### 5.1 `apps/web`

前端入口目前非常轻量，主要用于：

- 展示 `AiReport` 品牌与导航占位
- 告知开发者 DSL Schema、TypeScript 类型和 OpenAPI 的位置
- 作为后续工作台、AI 创建、模板中心、编辑器的承载入口

当前前端没有：

- 登录鉴权
- 项目列表数据请求
- AI 对话
- DSL 编辑器
- 报表运行时

也就是说，它现在是一个“工程骨架首页”，不是业务系统前端。

### 5.2 `packages/dsl-schema`

该模块定义了报表/大屏统一 DSL 的 JSON Schema，是整个项目的关键基础层。

DSL 顶层对象包含：

- `version`
- `type`
- `name`
- `description`
- `theme`
- `pages`
- `datasets`
- `parameters`

核心建模思想：

- 页面由 `pages` 组成
- 页面中包含多个 `widgets`
- 组件通过 `binding.datasetId` 绑定数据集
- 数据集通过 `sourceRef` 指向 SQL、数据表或 HTTP 源
- 组件支持筛选、排序、刷新策略、交互配置

当前支持的源类型：

- `sql`
- `table`
- `http`

当前支持的布局模式：

- `flow`
- `grid`
- `free`

当前支持的参数类型：

- `dateRange`
- `date`
- `string`
- `number`
- `enum`

该模块还提供：

- 样例 DSL：`sample-report.v1.json`
- 校验脚本：`scripts/validate-sample.mjs`

从工程质量上看，这部分已经具备继续扩展的基础。

### 5.3 `packages/dsl-types`

该模块提供与 Schema 对齐的 TypeScript 类型，用于：

- 前端编辑器使用强类型 DSL
- 服务端读写文档时获得类型约束
- AI 生成与修复逻辑的结构化输出建模

它与 JSON Schema 的关系是：

- Schema 是运行时约束
- TS 类型是开发时约束

这也是项目后续能否稳定演进的关键。

### 5.4 `services/api`

该模块计划承载平台服务端接口。当前实际情况如下：

已实现的路由：

- `POST /api/v1/ai/sessions`
- `POST /api/v1/ai/sessions/:id/messages`
- `GET /api/v1/projects`
- `POST /api/v1/projects`
- `GET /api/v1/projects/:id/document`
- `PUT /api/v1/projects/:id/document`
- `GET /api/v1/connections`
- `POST /api/v1/connections`
- `GET /api/v1/connections/:id/metadata`
- `GET /health`

已实现的中间件与基础能力：

- `helmet`
- `cors`
- JSON body 解析
- 请求日志
- 基于 Header 的租户/用户注入
- 统一错误处理

AI 接口当前只是占位逻辑：

- 会话可落库
- 消息可落库
- 流式返回通过 SSE 做了占位演示
- 真正的大模型调用与 DSL 生成尚未接入

项目接口当前支持：

- 创建项目
- 获取项目列表
- 读取 DSL 文档
- 保存 DSL 文档

数据源接口当前支持：

- 新建连接记录
- 查询连接列表
- 读取元数据快照

但后端模块目前有明显缺口：

- `app.ts` 中引用了 `query.ts`、`publish.ts`、`templates.ts`，对应文件并不存在
- API 依赖没有并入根工作区
- 在未安装 `services/api` 依赖时，`tsc` 无法通过
- OpenAPI 草案定义了更多接口，但代码未跟上

因此当前后端应被定义为“未完成的服务骨架”。

### 5.5 `database/migrations`

数据库脚本定义了较完整的业务域模型，包含：

- 租户与用户：`tenants`、`users`、`roles`、`user_roles`
- 数据源连接：`connections`
- 元数据缓存：`metadata_snapshots`
- 语义层：`semantic_models`、`measures`、`dimensions`、`relationships`
- 项目与文档：`projects`、`documents`
- 发布：`publishments`
- 模板：`templates`、`template_installations`
- AI 会话：`ai_sessions`、`ai_messages`
- 查询审计：`query_audit`

从表结构看，产品希望支持：

- 多租户隔离
- 数据源管理
- 报表/大屏 DSL 存储
- 模板中心
- 发布快照与回滚
- AI 会话审计
- 查询治理

`002_seed.sql` 提供了开发环境默认租户、用户和内置模板，用于配合 API 默认的 `DEFAULT_TENANT_ID` 与 `DEFAULT_USER_ID`。

### 5.6 `openapi/openapi.yaml`

OpenAPI 文件定义了平台的目标接口面。它覆盖的范围比当前服务端代码更完整，包含：

- AI 会话与消息
- 项目管理与文档保存
- 数据源连接与元数据读取
- 查询预览与批量查询
- 发布与只读渲染

因此可以把它理解为“目标 API 合同”，而不是“当前已实现清单”。

## 6. 运行方式

### 6.1 根仓库当前可运行部分

根仓库可直接使用的脚本：

```bash
npm install
npm run dev
npm run typecheck
npm run validate:dsl
```

当前根脚本的实际含义：

- `npm run dev`：启动前端 `@aireport/web`
- `npm run typecheck`：校验 `@aireport/dsl-types` 和 `@aireport/web`
- `npm run validate:dsl`：用 AJV 校验 DSL 样例

前端默认地址：

```text
http://localhost:5173
```

### 6.2 API 服务运行现状

理论上 `services/api` 应该独立安装依赖后运行：

```bash
cd services/api
npm install
npm run dev
```

但当前仓库里，API 服务还不能视为“开箱即用”，原因如下：

- 缺少 `query.ts`、`publish.ts`、`templates.ts`
- 根目录未管理其依赖安装
- 当前没有统一的根级启动脚本来同时启动 web + api

### 6.3 API 环境变量

`services/api/.env.example` 中定义了主要参数：

- `PORT`
- `DATABASE_URL`
- `REDIS_URL`
- `AI_PROVIDER`
- `AI_API_KEY`
- `AI_MODEL`
- `JWT_SECRET`
- `JWT_EXPIRES_IN`

此外，`env.ts` 中还支持：

- `DEFAULT_TENANT_ID`
- `DEFAULT_USER_ID`

这两个值用于开发环境下的租户与用户注入。

## 7. 代码与数据流关系

当前设计中的主数据流可概括为：

1. 用户通过前端发起“创建项目”或“AI 生成报表”请求。
2. API 服务将项目、AI 会话、消息等元数据写入 PostgreSQL。
3. AI 服务未来会根据元数据、模板、语义层和用户提示词生成 DSL。
4. DSL 文档以 JSONB 形式落入 `documents` 表。
5. 前端编辑器和运行时都围绕同一份 DSL 做渲染和保存。
6. 数据查询通过连接信息和数据集定义执行，并由 Redis 做缓存。
7. 发布后将 DSL 快照存入 `publishments`，供只读渲染使用。

这是一个典型的“DSL 中台化”结构。

## 8. 当前实现状态评估

### 8.1 已完成度较高

- 前端工程骨架可运行
- DSL Schema 定义完整
- DSL TypeScript 类型清晰
- DSL 样例校验通过
- PostgreSQL 数据模型较完整
- OpenAPI 草案覆盖面广
- 设计稿与开发方案齐备

### 8.2 已有实现但仍偏骨架

- API 服务目录与分层结构
- AI 会话路由
- 项目与文档路由
- 数据源连接路由
- 多租户请求上下文注入
- SSE 占位流式输出

### 8.3 主要缺口

- API 代码与 OpenAPI 草案不一致
- 后端缺少查询、发布、模板等关键路由实现
- 根工作区未包含 `services/api`
- 项目没有统一的一键启动开发体验
- 前端尚未接入真实接口
- AI 能力尚未接入真实模型与工具链
- 语义层目前只有数据库模型，没有服务实现
- 没有测试体系与 CI 配置

## 9. 当前验证结果

基于本地实际执行结果：

- 根仓库 `npm run typecheck`：通过
- 根仓库 `npm run validate:dsl`：通过
- `services/api` 的 `npm run typecheck`：失败
- `services/api` 的 `npm run build`：失败

失败原因主要有两类：

- 依赖未安装或未纳入根工作区
- 被引用的路由文件不存在

因此当前项目可以被定义为：

> 一个前端可启动、DSL 可校验、数据库模型较完整，但后端尚未闭环的 AI 报表平台原型仓库。

## 10. 建议的下一步开发顺序

如果以“尽快形成可演示 MVP”为目标，建议按以下顺序推进：

1. 将 `services/api` 正式纳入根工作区，补齐依赖安装路径。
2. 补全 `query`、`publish`、`templates` 三个缺失路由，并使 API 可以编译。
3. 打通前端工作台到项目列表、项目创建、DSL 读取/保存接口。
4. 让前端至少具备一个最小 DSL 预览能力。
5. 接入真实 AI Provider，先完成“自然语言 -> DSL 草稿”的最小闭环。
6. 补上统一开发脚本、测试与基础 CI。

## 11. 结论

`AiReport` 当前不是一个完整报表平台，而是一个方向明确、建模基础较扎实的原型工程。

它最有价值的部分不是现成功能，而是这三项基础设施已经被定义出来：

- 统一 DSL
- 较完整的数据模型
- 明确的产品与服务边界

如果后续继续开发，最优路径不是推翻重做，而是在现有 DSL、数据库模型和 API 草案之上，把服务实现与前端联调逐步补齐。
